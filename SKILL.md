# Skill 评估器

对 Skill 进行质量评估和多 Skill 横向对比。

## 使用场景

- 评审单个 Skill，发现不足和改进空间
- 对比两个或多个 Skill，判断哪个设计更优
- 在发布或分享前审计 Skill 质量
- 为 Skill 作者提供改进反馈

## 评估流程

### 第一步：加载目标 Skill

对每个待评估的 Skill：

1. 读取其 `SKILL.md` 文件
2. 列出所有附带资源（`scripts/`、`references/`、`assets/`）
3. 存在关键资源文件时，读取其内容以评估质量和相关性

若用户提供的 Skill 名称存在于 `~/.claude/skills/<name>/` 下，从该路径加载。若用户提供了文件路径，从指定路径加载。

### 第二步：按维度打分

从以下 **8 个维度** 对每个 Skill 打分，每个维度 0-10 分：

#### D1. 元数据质量（0-10）

评估 YAML frontmatter 中的 `name` 和 `description`：

| 分数 | 标准 |
|------|------|
| 0-3 | 缺失、模糊或泛泛而谈（如 "Replace with description"） |
| 4-6 | 描述了 Skill 的功能，但缺少触发条件 |
| 7-8 | 清晰描述功能并包含使用场景 |
| 9-10 | 精确描述，同时包含正向触发条件和反向排除条件（如 "TRIGGER when... DO NOT TRIGGER when..."） |

#### D2. 执行引导清晰度（0-10）

Skill 被加载后，Agent 是否清楚该怎么执行？面对不同输入能否正确决策？

| 分数 | 标准 |
|------|------|
| 0-3 | 加载后 Agent 仍不知道该做什么，缺少具体指令 |
| 4-6 | 有基本执行步骤，但面对不同用户输入时缺少分支引导 |
| 7-8 | 对不同场景有清晰的执行路径，说明了何时需要追问用户 |
| 9-10 | 执行引导完备：覆盖多种输入场景的分支逻辑，明确执行边界（该做什么、不该做什么），包含异常情况的处理指引 |

#### D3. 领域知识密度（0-10）

Skill 是否内嵌了通用 Agent 不具备的专业知识？

| 分数 | 标准 |
|------|------|
| 0-3 | 无专业知识，只是包装了基础指令 |
| 4-6 | 包含一些领域特定的指导或约定 |
| 7-8 | 有丰富的专家知识、数据模型或最佳实践 |
| 9-10 | 深度的、难以获取的领域专业知识，包含细致的决策逻辑 |

#### D4. 工作流完整性（0-10）

Skill 是否定义了清晰的端到端工作流？

| 分数 | 标准 |
|------|------|
| 0-3 | 无工作流，只是单个操作或模糊指令 |
| 4-6 | 列出了部分步骤，但缺少异常处理或分支逻辑 |
| 7-8 | 完整的多步骤工作流，步骤顺序清晰 |
| 9-10 | 健壮的工作流，包含分支、异常处理和迭代循环 |

#### D5. 输入输出清晰度（0-10）

用户是否知道该提供什么、能得到什么？

| 分数 | 标准 |
|------|------|
| 0-3 | 输入和输出均未描述 |
| 4-6 | 部分描述，用户需自行推断 |
| 7-8 | 清晰的输入要求和输出格式，附有示例 |
| 9-10 | 明确的输入输出契约，包含多个示例和边界情况 |

#### D6. 资源利用（0-10）

Skill 对附带资源（scripts、references、assets）的利用程度如何？

| 分数 | 标准 |
|------|------|
| 0-3 | 明显需要资源却未提供任何附带资源 |
| 4-6 | 有部分资源但利用不足或组织混乱 |
| 7-8 | 资源利用合理：脚本用于确定性任务，references 用于领域文档 |
| 9-10 | 资源设计优秀，遵循渐进式披露；SKILL.md 精简，配套文件丰富 |
| N/A | Skill 确实不需要附带资源（按 7 分基线计分） |

#### D7. 写作质量（0-10）

SKILL.md 结构是否清晰、是否易于 Agent 理解和执行？

| 分数 | 标准 |
|------|------|
| 0-3 | 组织混乱、表述模糊或指令自相矛盾 |
| 4-6 | 可读但结构或简洁性有待提升 |
| 7-8 | 组织良好，标题清晰，祈使语气，逻辑流畅 |
| 9-10 | 优秀的技术写作：可扫读、精确、无冗余 |

#### D8. 范围与聚焦（0-10）

Skill 是否专注解决一类问题，既不过宽也不过窄？

| 分数 | 标准 |
|------|------|
| 0-3 | 过于宽泛（试图包揽一切）或过于琐碎（不值得做成 Skill） |
| 4-6 | 范围合理但存在功能蔓延或缺少边界定义 |
| 7-8 | 聚焦于单一且连贯的问题域 |
| 9-10 | 范围精确，明确说明了哪些不在范围内 |

### 第三步：计算综合得分

按加权求和计算总分：

| 维度 | 权重 |
|------|------|
| D1. 元数据质量 | 15% |
| D2. 执行引导清晰度 | 15% |
| D3. 领域知识密度 | 15% |
| D4. 工作流完整性 | 15% |
| D5. 输入输出清晰度 | 10% |
| D6. 资源利用 | 10% |
| D7. 写作质量 | 10% |
| D8. 范围与聚焦 | 10% |

**综合得分 = sum(各维度分数 × 对应权重)**

等级映射：

| 分数 | 等级 | 含义 |
|------|------|------|
| 9.0-10 | S | 卓越 — 标杆级 Skill |
| 7.5-8.9 | A | 优秀 — 可直接使用，仅需少量打磨 |
| 6.0-7.4 | B | 合格 — 可用但有明显改进空间 |
| 4.0-5.9 | C | 较弱 — 需要大幅修改 |
| 0-3.9 | D | 较差 — 存在根本性问题，建议重新设计 |

### 第四步：列出问题并给出改进建议

对每个低于 7 分的维度，提供：

1. **问题**：具体缺少什么
2. **影响**：为什么这很重要（如 "Agent 在相关场景下无法触发该 Skill"）
3. **建议**：具体可操作的改进方案，附示例

### 第五步：多 Skill 对比（提供多个 Skill 时）

评估 2 个及以上 Skill 时，输出对比分析：

#### 5a. 并列评分表

```
| 维度             | Skill A | Skill B | Skill C |
|-----------------|---------|---------|---------|
| D1. 元数据质量      |   8     |   4     |   6     |
| ...             |  ...    |  ...    |  ...    |
| 综合得分         | 7.8/A   | 4.2/C   | 6.5/B   |
```

#### 5b. 各自优劣势

对每个 Skill 说明：
- 相比其他 Skill **更强**的方面
- 相比其他 Skill **更弱**的方面

#### 5c. 经验总结

识别共性规律：
- 得分最高的 Skill 做对了什么，值得其他 Skill 借鉴
- 所有被评估 Skill 的共同短板

#### 5d. 排名

从优到劣排列所有 Skill，每个位次附一句理由。

## 输出格式

评估报告按以下结构组织：

```
## Skill 评估报告

### [Skill 名称]

**综合得分：X.X / 10（等级 X）**

| 维度 | 分数 | 说明 |
|------|------|------|
| D1. 元数据质量 | X/10 | ... |
| ... | ... | ... |

#### 问题与建议
1. **[维度名]**：[问题] → [建议]
2. ...

---
（对每个 Skill 重复以上内容）

### 对比分析（多个 Skill 时）
（并列评分表、优劣势、经验总结、排名）
```

## 注意事项

- 保持客观，以证据为基础；打分时引用 Skill 中的具体行或章节
- 简单但执行到位的 Skill 同样可以得高分 — 复杂性不是优点
- 并非所有 Skill 都需要附带资源；D6 的打分应基于资源是否真正有帮助
- 当用户未指定评估哪些 Skill 时，主动询问 Skill 名称或路径
- 语言适配：评估报告的语言应与目标 Skill 的主要语言一致。若目标 Skill 为英文，则报告、维度名称和建议均使用英文输出；若用户明确指定了输出语言，以用户指定为准
